# 社区点赞业务缓存设计优化探索

> 来源：https://xie.infoq.cn/article/f6840380238de0761abe39e08
> 记录时间：2026-07-01

---

## 核心观点

这篇文章最关键的启发，不是“Redis 用 Set 还是 Hash”这种局部技巧，而是：**缓存建模必须跟真实访问模式对齐**。当上游主场景已经从“查一篇内容的点赞状态”变成“一个用户在 Feed 里批量判断几十篇内容是否点赞”时，继续按内容维度缓存全量点赞用户，只会把 Redis 请求、TTL 维护和热点 key 压力不断放大。最终更有效的方向，是把状态缓存切到**用户维度**，把点赞总数拆给独立计数服务，再用冷热边界控制缓存范围。

---

## 一、文章到底讲了什么

文章回顾了社区点赞缓存方案从 v1 到 v4 的四次演进，核心目标一直围绕三件事：

1. **高性能**：用户刷 Feed 时，需要高频判断“是否点赞”
2. **高稳定性**：热点内容不能把缓存击穿、也不能把 Redis 打抖
3. **低成本**：不能为了兜住热点，把 Redis 空间和 DB 回查成本无限做大

它给出的结论非常典型：

- v1 主要输在 **Key/Value + JSON 解析** 过重
- v2 解决了 JSON 解析问题，但引入了 **大 Key**
- v3 通过 **分片 Set** 缓解大 Key，但仍然没有解决 Feed 批量判断放大问题
- v4 才真正把建模从“内容中心”切成“用户中心”

文中给出的最终收益也很直观：

- Redis 查询 QPS 峰值下降 **20 倍**
- 接口平均 RT 下降 **10 倍**
- DB 查询 QPS 峰值下降 **6 倍**
- 缓存节省约 **16G** 存储空间

## 二、四次演进脉络

### v1.0：KV + JSON

最早的思路是：

- 每篇内容一个 Redis Key
- value 是该内容所有点赞用户 ID 的 JSON 数组

类似这样：

```text
cId => "[uid1,uid2,uid3...]"
```

查询路径：

1. 从 Redis 取整段 value
2. 反序列化成数组
3. 用 `in_array` 判断是否点赞
4. 用 `count` 计算点赞数

问题非常集中：

- 重建缓存要回查整篇内容所有点赞用户，容易慢 SQL
- 反序列化和数组扫描开销大
- 每次点赞/取消赞都直接删缓存，热点下会发生缓存击穿

### v2.0：内容维度 Set

v2 的改造方向很自然：

- Redis 不再存 JSON，而是直接用集合
- 一个内容一个 Set
- `SISMEMBER` 判断是否点赞
- `SCARD` 获取点赞数

这一步解决了两个明显问题：

- 不再需要每次取全量数据再反序列化
- 点赞状态和计数查询都可以走 Redis 原生操作

但它也带来了新的结构性问题：

- 热点内容的点赞用户非常多，Set 很容易变成 **大 Key**
- 热点 key 的过期与清理，对 Redis 稳定性不友好
- 只要缓存需要重建，仍然要回查全量点赞用户

### v3.0：内容维度分片 Set

为了避免单个大 Key，v3 把同一篇内容的点赞用户按分片拆开：

```text
cid_slice1 => [uid1,uid11,uid111...]
cid_slice2 => [uid2,uid22,uid222...]
```

同时点赞总数交给独立计数服务维护，不再每次都依赖 `SCARD`。

这一步能缓解：

- 大 Key 风险
- 单个 key 的维护成本
- `SCARD` 这类集合统计消耗

但它还是没有解决一个更本质的问题：

> **Feed 场景是“一个用户查一批内容”，而不是“一篇内容查一批用户”。**

只要缓存还是按内容组织，用户刷一次 Feed，就要对每篇内容分别查一次状态。请求量虽然分片了，但**访问模式仍然被放大**。

### v4.0：用户维度 Hash

v4 才是文章真正的关键转向：

- key 变成 `user:{uid}:likes`
- field 记录最近点赞过的 `contentId`
- Hash 中额外维护：
  - `ttl`：逻辑过期时间标记
  - `minCid`：冷热边界
- 内容点赞总数拆到独立计数服务

文中的结构大意如下：

```json
{
  "userId": {
    "ttl": 1653532653,
    "cid1": 1,
    "cid2": 1,
    "cidn": 1,
    "minCid": 3540575
  }
}
```

这个设计一次解决了多个问题：

1. **批量判断更顺手**
   - 同一个用户看一批内容时，状态都在一个 user hash 里
2. **状态与计数拆开**
   - “是否点赞” 和 “点赞总数” 不再塞进同一份缓存模型
3. **冷热数据可控**
   - 只维护近期内容，老内容不再长期占住缓存
4. **避免频繁 TTL 查询**
   - 用逻辑字段维护有效期，而不是每次都额外打一遍 `TTL`

## 三、为什么这个专题 demo 只实现 v2 和 v4

如果把 v1~v4 全部都写成 runnable，会让重点被冲散。

这一版 demo 只实现：

- **v2：内容维度 Set**
- **v4：用户维度 Hash**

原因是它们最能代表文章里最重要的一次认知升级：

- v2 代表“继续按内容维度建模，但尽量用 Redis 原生结构优化”
- v4 代表“直接按用户维度重建缓存模型，匹配 Feed 批量查询方式”

而 v3 更像是 v2 路线上一次必要但过渡性的工程优化：它缓解了大 Key，却没有彻底解决访问模式不匹配的问题。

## 四、Demo 设计

本目录下的 demo 不是要做压测框架，而是做一个**结构性对比实验**：

- 用同一组点赞事件驱动两套缓存模型
- 对比它们在批量 Feed 判断中的命令数量与热点分布
- 观察为什么 v4 的思路在信息流场景下更合适

### 目录结构

```text
likecache/
├── README.md
├── go.mod
├── cmd/
│   └── compare/
│       └── main.go
└── internal/
    └── store/
        ├── types.go
        ├── v2_set.go
        └── v4_hash.go
```

### v2 demo 关注什么

- `content:{cid}:likes -> Set(userID)`
- `SISMEMBER` 判断状态
- `SCARD` 获取点赞数
- 每篇内容都要单独查一次状态
- 热点内容读写时还会顺带维护 TTL

### v4 demo 关注什么

- `user:{uid}:likes -> Hash(contentID=1, ttl=..., minCid=...)`
- 同一个用户的一批内容状态集中存放
- 点赞总数从 user hash 中拆开，单独走 counter
- `minCid` 用于表达“只缓存一段时间内的最近内容”

## 五、运行方式

在当前目录执行：

```bash
go run ./cmd/compare
```

程序会输出：

- 两种缓存模型的结构说明
- 同一组 Feed 请求下的点赞状态判断结果
- 点赞/取消赞后的计数检查
- 对比摘要：`readOps / writeOps / ttlOps / totalOps / keyCount / coldPathFallbacks`

---

## 六、如何理解输出

### 如果是 v2

你会看到这类特征：

- 每批 Feed 判断都会对每篇内容各查一次状态
- `ttlOps` 明显增加
- 热点集中在内容维度 key 上

这正对应文章里 v2 / v3 路线的瓶颈：

- 内容越热，单 key 越重
- Feed 一次看几十篇内容，读路径就会被放大几十倍

### 如果是 v4

你会看到这类特征：

- 一次批量判断更接近“按用户读一次”
- `ttlOps` 近似为 0（由逻辑字段替代）
- 热点不再集中在单个 content Set，而是分散在活跃用户的 Hash 上
- 会出现 `coldPathFallbacks`，表示老内容被踢出热缓存范围

这对应文章里最终方案的设计目标：

- 批量状态判断不再被“内容个数”线性放大
- 老内容缓存命中率低时，不再继续消耗热缓存空间

---

## 七、适用边界

这个方案并不是说“Hash 永远比 Set 好”，而是说：

> **当主场景是“一个用户看一批内容是否点赞”时，用户维度建模更贴近真实访问模式。**

如果你的核心访问模式换成：

- “给我这篇内容最近点赞的用户列表”
- “围绕某篇超级热点内容做聚合分析”

那内容维度结构依然可能更直接。

所以文章真正值得记住的是：

- 先看**查询模式**
- 再决定**缓存维度**
- 最后才是选 **Set / Hash / 分片 / TTL 技巧**

## 八、可以继续扩展的方向

如果后面想把这个专题继续做深，可以补：

1. **真实 Redis 接入版**
   - 用 `go-redis` 直接执行 `SADD/SISMEMBER/SCARD/HSET/HMGET`
2. **v3 分片 Set 版**
   - 把“缓解大 Key 但没解决访问模式”也做成代码实验
3. **压测版 workload**
   - 随机生成大规模用户 / 内容 / 热点流量，输出更完整的统计报告
4. **计数服务拆分版**
   - 把 like count 抽象成单独接口，更接近文章架构
