点赞缓存
为什么要从 Set 改成 Hash
Feed 场景下，真正重要的不是命令技巧，
而是缓存维度有没有对齐真实查询路径。
旧方案
内容维度 Set
每篇内容都维护一个点赞集合。
用户刷一次 Feed，要按内容逐个判断状态。
新方案
用户维度 Hash
一个用户最近点赞过的内容集中在一个 Hash。
批量状态查询更贴近信息流访问模式。
1
热点内容容易变大 Key
按内容存全量点赞用户，热点越高，集合越重。
2
Feed 会把读路径放大
一刷就是多篇内容，单篇逐查会让 Redis 请求数快速膨胀。
3
状态和计数要拆开
是否点赞与点赞总数不是同一类数据，分开维护更清晰。
4
冷热边界更重要
老内容不应长期占据高价值缓存，热数据要更聚焦。
