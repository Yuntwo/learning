## 正文版本（推荐）

最近整理了一篇关于**社区点赞缓存设计优化**的文章，最有启发的一点不是“Redis 用 Set 还是 Hash”，而是：**缓存建模一定要跟真实访问模式对齐。**

很多点赞系统一开始都会这么做：
- 按内容维度缓存所有点赞用户
- 判断是否点赞时查内容的点赞集合
- 点赞数直接从集合里统计

这样做在早期很自然，但到了 Feed 场景会越来越吃力：
- 用户刷一次信息流，要批量判断很多内容是否点赞
- 每篇内容都要单独查一次状态
- 热点内容容易出现大 Key
- TTL 维护、缓存重建、回源压力都会被放大

这篇文章里最值得记的 4 次演进：

### 1）v1：KV + JSON
把某篇内容所有点赞用户存成 JSON。
问题是反序列化重、回查全量点赞用户成本高，而且一删缓存就容易击穿。

### 2）v2：内容维度 Set
改成 Set 后，是否点赞可以直接做成员判断，点赞数也能直接统计。
比 JSON 好很多，但热点内容还是会形成大集合，热点 key 压力依旧明显。

### 3）v3：分片 Set
把一篇内容的点赞用户拆成多个分片，缓解大 Key。
但本质上还是“按内容维度查状态”，用户刷 Feed 时依然容易把请求量放大。

### 4）v4：用户维度 Hash
真正关键的一步，是把状态缓存从“按内容存用户”改成“按用户存最近点赞内容”。
这样当一个用户看一批内容时，状态判断就更接近一次按用户维度的批量读取。

这类设计还会顺手解决几个问题：
- 状态与计数拆开，职责更清晰
- 热点不再集中砸到单个 content key
- 可以用冷热边界控制缓存范围
- 老内容不再长期占住高价值缓存空间

我自己顺手做了个小 demo，对比内容维度 Set 和用户维度 Hash 的结构差异。最直观的感受是：

**真正该先想清楚的不是 Redis 命令，而是：你的主查询路径到底是“按内容查”，还是“按用户批量查”。**

如果主场景已经是信息流批量判断，那缓存维度选错了，后面再怎么补 TTL、分片、局部优化，都只是延缓问题暴露。

一句话总结：

> **缓存设计先看访问模式，再看数据结构。维度选对，优化才有意义。**

## 正文版本（短版）

最近整理社区点赞缓存设计时，最大的感受是：
**缓存优化的重点往往不是“Set 还是 Hash”，而是缓存维度有没有和真实访问模式对齐。**

如果系统主场景已经变成：
- 一个用户刷 Feed
- 要批量判断很多内容是否点赞

那继续按“内容维度”缓存全量点赞用户，就会带来：
- 每篇内容都要单独查状态
- 热点内容变成大 Key
- TTL 和缓存维护成本不断放大

更合理的方向是：
- 状态缓存按用户维度组织
- 点赞总数拆给独立计数
- 用冷热边界控制缓存范围

一句话：
**先看查询模式，再决定缓存维度，最后才是 Set / Hash / 分片这些实现细节。**
