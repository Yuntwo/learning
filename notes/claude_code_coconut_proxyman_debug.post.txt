Claude Code 直连模型不通时，我最近用了一套很实用的本地调试链路：本地模型代理 + 抓包代理分层。

核心不是把两个端口改成一样，而是把流量顺序走对。

我的做法是：本地模型代理监听 8787，Claude Code 的 ANTHROPIC_BASE_URL 指向它；抓包工具保持监听 9090，再让 Claude Code 的 HTTP_PROXY / HTTPS_PROXY 指向 9090。

这样链路就变成：Claude Code → 抓包代理 → 本地模型代理 → 目标模型服务。

几个很关键的坑：
不要把 127.0.0.1 / localhost 放进 NO_PROXY，不然本地 8787 会直接绕过抓包代理。
抓 HTTPS 时记得信任抓包工具证书，必要时补 NODE_EXTRA_CA_CERTS。
如果模型调用成功但抓不到包，十有八九是代理变量没有真正生效，或者被 NO_PROXY 绕过了。

一句话总结：模型转发端口负责“处理请求”，抓包端口负责“观察请求”，两者不用相同，只要流量先经过抓包代理就行。

Claude Code 模型调试链路

[小红书正文]
如果 Claude Code 直连模型不通，一个很好用的办法是本地挂一个兼容代理，再在前面串一个抓包代理做观测。

我这次踩完坑后的核心结论是：抓包时，不需要把抓包工具的端口改成和模型代理一样。

真正关键的是链路顺序。

正确思路是：
Claude Code 先走抓包代理
抓包代理再转发到本地模型代理
本地模型代理再去请求目标模型服务

如果本地模型代理在 8787，抓包代理在 9090，那么常见配置就是：
ANTHROPIC_BASE_URL 指向 127.0.0.1:8787
HTTP_PROXY / HTTPS_PROXY 指向 127.0.0.1:9090

这里最容易忽略的是 NO_PROXY。
如果你把 127.0.0.1 或 localhost 放进去了，本地 8787 这段通常会直接绕过抓包代理。表现就是模型能正常返回，但你在抓包工具里什么也看不到。

另一个常见问题是 HTTPS 证书。
如果想看明文请求和响应，只改代理端口还不够，还要让系统信任抓包工具证书。必要时给 Node 进程补一个 NODE_EXTRA_CA_CERTS。

我的体感是，这套方法特别适合做三件事：
排查模型请求到底有没有发出去
确认请求是否真的经过本地代理
观察输入输出和报错细节，快速定位链路问题

一句话总结：
模型代理负责把请求转出去，抓包代理负责把链路看清楚。两者分层，调试效率会高很多。
