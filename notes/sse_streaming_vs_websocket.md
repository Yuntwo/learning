# SSE 与流式输出入门：格式、优缺点与适用场景

> 来源：基于与 Claude 的对话整理
> 记录时间：2026-07-22

---

## 核心观点

SSE（Server-Sent Events）是一种基于 HTTP 的服务端单向推送协议，特别适合 AI 对话流式输出、实时日志、状态更新这类“服务端不断往前端推文本事件”的场景；如果需要真正双向实时通信，则更适合 WebSocket。

---

## 要点整理

### 1. 什么是流式输出

流式输出（streaming）指的是：服务端不等整条结果完全生成完毕，而是把生成中的内容分片、按顺序持续返回给客户端。

典型体验是：

- 页面出现“打字机”效果
- 用户能更早看到首字节
- 长内容无需等待整段生成完毕

在大模型场景里，流式输出最常见的表现就是 token 或文本片段持续到达，前端一边接收，一边拼接显示。

### 2. SSE 是什么

SSE 是 **Server-Sent Events** 的缩写，中文通常可以理解为“服务端发送事件”。

它建立在普通 HTTP 连接之上，但响应不是一次性结束，而是使用 `text/event-stream` 持续向客户端推送事件。

常见响应头：

```http
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

常见事件格式：

```text
event: message_start
data: {"type":"message_start", ...}

event: content_block_delta
data: {"type":"content_block_delta","delta":{"text":"在"}}

event: content_block_delta
data: {"type":"content_block_delta","delta":{"text":"。"}}

event: message_stop
data: {"type":"message_stop"}
```

其中：

- `event:` 表示事件类型
- `data:` 表示该事件携带的数据
- 事件之间通常用空行分隔
- 客户端按顺序读取并处理每个事件

### 3. 你看到的流式输出怎么还原为最终文本

像下面这样的多段：

```text
"在"
"。"
"有什么"
"我"
"可以"
"帮"
"你的"
"？"
```

客户端最终会把它们按顺序拼起来，得到：

```text
在。有什么我可以帮你的？
```

这就是典型的“增量输出 + 客户端拼接”。

### 4. SSE 与“普通非流式响应”的区别

| 维度 | SSE / 流式输出 | 普通非流式响应 |
|------|----------------|----------------|
| 返回方式 | 边生成边返回 | 全部生成完再返回 |
| 用户体验 | 更快看到内容 | 首次等待更久 |
| 适合内容 | 长文本、日志、事件流 | 短响应、一次性结果 |
| 客户端处理 | 需要增量解析 | 直接读完整响应 |
| 实现复杂度 | 稍高 | 较低 |

### 5. SSE 和 WebSocket 的区别

这两者经常一起被提到，但适用目标并不一样。

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 全称 | Server-Sent Events | WebSocket |
| 通信方向 | 服务端 → 客户端 | 客户端 ↔ 服务端 |
| 协议基础 | HTTP | 独立升级协议 |
| 数据形式 | 文本事件流为主 | 文本和二进制都适合 |
| 浏览器接入 | 简单，原生 `EventSource` | 需要 `WebSocket` 对象 |
| 断线重连 | 通常更容易处理 | 需要自行管理较多状态 |
| 适合场景 | AI 输出、通知、日志、进度 | 聊天、协作编辑、游戏、双向指令 |

### 6. SSE、WebSocket、普通 HTTP 的格式、优缺点、场景对比

| 方案 | 典型格式 | 优点 | 缺点 | 适合场景 |
|------|----------|------|------|----------|
| 普通 HTTP 响应 | 一次性 JSON / HTML | 简单、稳定、好调试 | 需要等结果完整返回 | 查询接口、表单提交、静态内容 |
| SSE | `event:` + `data:` 的事件流 | 接入简单、适合文本增量、天然贴合 AI streaming | 基本是单向推送，不适合频繁双向交互 | AI 对话、日志流、任务进度、通知 |
| WebSocket | 长连接消息帧 | 真正双向实时通信、实时性强 | 连接管理更复杂、状态维护更多 | 即时聊天、多人协作、实时控制、游戏 |

### 7. 为什么大模型接口常用 SSE

AI 场景里，服务端生成文本通常是“逐 token 产生”的，自然适合以事件流方式持续推送。

SSE 在这里的优势很明显：

- 前端容易做“打字机效果”
- 用户能更早看到结果，等待感更低
- 对“文本增量”这种输出形态非常自然
- 基于 HTTP，代理、网关、服务端框架普遍更容易接入

所以很多大模型 API 在开启 stream 模式后，都会返回类似下面的结构：

```text
event: content_block_delta
data: {"delta":{"text":"hello"}}
```

### 8. 什么时候不该优先选 SSE

如果你的需求不是“服务端单向持续推消息”，而是下面这类情况，SSE 就不是最优：

- 前端需要频繁向服务端发实时指令
- 双方都要保持高频低延迟双向通信
- 需要传输大量二进制数据
- 连接状态、会话同步、协同动作非常复杂

这时更适合 WebSocket。

## 快速判断：应该用哪种方式

可以用一个简单判断法：

| 如果你的需求是… | 更适合 |
|------------------|--------|
| 查一次结果，拿完整响应就结束 | 普通 HTTP |
| 服务端不断吐文本、日志、状态给前端 | SSE |
| 前后端都要实时互发消息 | WebSocket |

一句话记忆：

- **一次性返回：HTTP**
- **单向持续推送：SSE**
- **双向实时通信：WebSocket**

## 最佳实践 / 注意事项

### 服务端侧

- 正确设置 `Content-Type: text/event-stream`
- 保持连接不断开，按事件分块发送
- 每个事件结束后写空行，保证客户端能正确分帧
- 长连接场景下注意超时、代理层缓冲和心跳保活

### 客户端侧

- 把 `delta` 事件按顺序拼接，不要覆盖旧内容
- 正确处理 `message_start`、`message_stop`、错误事件和中断事件
- UI 上要区分“还在生成”和“已经结束”
- 如果是通用网络层，最好把“事件解析”和“文本拼接”分层实现

### 产品体验侧

- 能流式就尽量流式，尤其是长文本生成
- 如果结果非常短，是否流式要看实现收益是否值得
- 对用户来说，“尽快看到第一段内容”通常比“总耗时少一点点”更重要

---

## 个人理解总结

SSE 的价值不在于“技术更高级”，而在于它刚好匹配“服务端连续产出文本、客户端持续消费”的问题形态。所以在 AI、日志、通知、进度流这类场景里，它往往比普通 HTTP 更自然；但一旦需求升级成双向实时协作，WebSocket 通常更合适。
