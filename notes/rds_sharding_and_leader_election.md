## 核心观点

一个 RDS「数据库」是逻辑概念,底下叠了两个**正交**的维度:**分库分表**(切数据,各片不同)和**主从复制**(复制数据,各副本相同)。代理(Proxy)/SDK 负责把 SQL 路由到正确的分片和读/写实例;而主从的高可用,本质上依赖一套「故障检测 + 领导选举(选主)」机制——这正是分布式系统通用课题在数据库里的落地。

---

## 1. 库 / 代理 / 实例 / 分片 / 主从 的关系

```
数据库(逻辑库,业务视角的一个库,一个连接入口)
 └─ Proxy 集群(无状态路由层,可水平扩展)
      └─ 分库分表 → 切成 N 个分片(Shard)
           ├─ 分片0 ──→ 复制组:1 主 + N 从
           ├─ 分片1 ──→ 复制组:1 主 + N 从
           └─ 分片N ──→ 复制组:1 主 + N 从
```

| 层级 | 含义 | 数量关系 |
|------|------|----------|
| 数据库(逻辑库) | 业务视角的一个库 | 1 |
| Proxy(代理) | 无状态路由:分片路由 / 读写分离 / 连接收敛 | N 个,与分片数无强绑定,按流量扩 |
| 分片(Shard) | 数据按分片键水平切分后的逻辑单元 | M 个 |
| 实例(Instance) | 真正跑 MySQL 的进程/容器 | ≈ 分片数 ×(1 + 从库数) |
| 写库(主) | 每个分片唯一 Master,承接写和强一致读 | 每分片 1 个 |
| 读库(从) | 每个分片的 Slave,binlog 复制,承接只读 | 每分片若干 |

**关键:**
- **Proxy 数 ≠ 分片数**。Proxy 是无状态路由,横向扩为扛连接/QPS;分片为拆数据。各自独立伸缩。
- **一个分片 = 一个 MySQL 复制组(1 主 N 从)**。
- 写只走主,读可走主或从。

## 2. 读写路由谁决定?代理是必须的吗?

三个「选择」是不同层次的决策,不一定都在 Proxy:

| 决策 | 默认在哪 | 能否下放/被覆盖 |
|------|---------|----------------|
| ① 选哪个分片 | Proxy 或客户端 SDK 按分片键算 | 这一步是分库分表本质,跑不掉 |
| ② 走主还是从 | Proxy/SDK 默认策略 | **业务可强制覆盖**:事务内读走主、`/*+ master */` hint、写后强一致读 |
| ③ 选分片内哪个从库 | Proxy/SDK 的 LB(轮询/随机/最小延迟/权重)+ 延迟过大自动摘除 | 最容易下沉到 DNS/VIP/四层 LB |

**代理不是必须的。** 两种主流形态:
- **Proxy 模式**:App→Proxy 集群→后端主从。优点:业务无侵入、多语言、**连接收敛**、集中运维;缺点:多一跳、Proxy 自身要做 HA。代表:字节 RDS Proxy、ProxySQL、Vitess VTGate。
- **SDK 直连模式**:分库分表 SDK 嵌在应用里,本地算分片+读写分离,直连后端。优点:无中间跳、无单点;缺点:语言绑定、**连接数爆炸**、配置变更要推全量客户端。代表:ShardingSphere-JDBC、TDDL。

大厂偏向 Proxy 的核心理由往往是**连接收敛**(后端 MySQL 连接数稀缺)而非省延迟。

## 3. 分库分表 vs 主从 vs 分片(易混淆)

两个正交维度:

| 维度 | 解决什么 | 数据是否相同 |
|------|---------|------------|
| 分库分表(Sharding) | 数据太多,水平拆开 | 各片**不同**(拼起来才是全量) |
| 主从(Replication) | 高可用 + 读扩展,同数据复制多份 | 各副本**相同** |

一句话:**分库分表是「切蛋糕」(每片不一样),主从是「复印」(每份一样)。**

- 分表:一张大表拆成多张同构表(`order_0~order_15`)
- 分库:这些表分散到多个物理库/实例
- 分片(Shard):拆分后的逻辑单元统称,是**分库分表的产物**,不是和它并列的概念
- 所以:`分库分表 → 产生分片;每个分片内部 → 用主从`。分库分表 ≠ 主从。

## 4. 读库写库可能都是 3306 吗?怎么区分?

**可以,而且通常都是 3306。** 主从是独立 MySQL 进程,各跑自己 host 的 3306,区别在 **IP/host 不同**,不靠端口。

- 区分读写的本质:**这条 SQL 该发给哪个 host**,不是哪个端口。
- Proxy/SDK 靠**管控层维护的拓扑元数据**知道谁是主谁是从:
  ```
  shard_0:  master = 10.1.1.1:3306
            slaves = [10.1.1.2:3306, 10.1.1.3:3306]
  ```
  启动时拉取并订阅变更;主从切换时管控层更新元数据并推送,路由刷新——端口不变,master 角色指向的 IP 变。
- 常见落地:Proxy 分流 / 双域名(`xxx-write.db`、`xxx-read.db`)/(少见)一机多实例用 3306/3307 区分**实例**,但端口仍不代表读写角色。

## 5. 逻辑分片 vs 物理分片(实践案例)

监控页指标的分层(案例:101 逻辑分片,`master_count`=`shardingNum`=12,某实例 `sharding_no`=50~57):

| 指标 | 值 | 层 | 含义 |
|------|----|----|------|
| 逻辑分片数 | 101 | 逻辑层(应用路由) | 应用 `hash(分片键) % 101` 映射到 0~100 逻辑桶 |
| shardingNum | 12 | 物理层 | 实际物理分片数 |
| master_count | 12 | 物理层 | 12 个 master,1 物理分片 = 1 复制组 |
| sharding_no | 50~57 | 映射 | 该实例承载逻辑分片 50~57 这一段(8 个) |

**结论:101 只是逻辑分片;物理上只有 12 个分片/master。** 每台物理实例打包承载一段连续逻辑分片(101÷12≈8.4,故 `sharding_no` 是 50~57 这种区间)。

**为什么逻辑数 > 物理数(解耦应用路由与物理拓扑):**
1. 应用路由永远 `% 101` 不变,完全不感知落在几台机器。
2. 扩缩容不动哈希:把某些逻辑分片整段搬到新实例即可,无需重新取模/洗牌——可按逻辑分片粒度做在线迁移与再均衡。
3. 粒度更细的负载均衡,热点可拆。
- Proxy/SDK 里有「逻辑分片号→物理实例」映射表;扩容=改表+迁移对应逻辑分片,应用无感。(映射表在 RDS 控制台「分片拓扑/路由规则」,不在 Grafana。)

## 6. 故障检测与领导选举(源文章核心,支撑主从切换)

主从的「选主/failover」本质是分布式系统的**领导选举**问题。

**为什么要有领导者:** 稳定的 leader 能减少远程节点状态同步消耗、减少消息数,部分操作在单一 leader 内完成避免集群同步。代价是 leader 可能成为单点瓶颈——解法是**分区,每个分区有自己的 leader**(如 Spanner;也对应 RDS 里每个分片各自一套主从)。

**选举算法:**
- **Bully 算法**(经典,易收敛):用节点 ID 大小选主,ID 最大者当选。4 步:查更大 ID 节点→无则发 Victory 宣告为主;有则发 Election 等响应→超时未回则自己当主并向更小 ID 发 Victory;收到更小 ID 的 Election 则回 Alive。
  - 缺陷:网络分区下违反「安全性」(可能选出多个 leader);对高 ID 节点有强偏好,不稳定高 ID 节点会反复连任→可结合存活性指标评估活性来缓解。
- **Bully 改进版**:① 故障转移节点列表(leader 预先给候选列表,缩短重选时间);② 节点分角色(候选/普通,减少消息数,随机启动延迟避免并发选举);③ 邀请算法(节点组合并,分而治之);④ 环形算法(节点成环,选举消息沿环传递取最大 ID)。
- **ZAB / Multi-Paxos / RAFT**:不止做选举,还同时解决故障检测和竞争冲突,内涵远大于单纯选举(共识算法,后续专讲)。

**脑裂(split-brain)与法定人数(Quorum):**
- 上述算法都可能脑裂——分区后两个子网各自选出一个互不知情的 leader。
- 解法:**引入最小法定人数选主**。例:5 节点,最小法定人数=3。网络断开后,C/D/E 满足 3 个→选出新 leader 正常工作;A/B 只有 2 个<3→A 主动放弃 leader,该子网标记不可用拒绝服务,从而避免脑裂。(Elasticsearch 即 Bully + 最小法定人数。)
- **选举必须和故障检测结合**:集群需及时感知 leader 是否仍活跃(基于超时的故障检测),leader 崩溃/不可达时才触发重选。

**与第 14 讲(错误侦测)的衔接:** 故障检测是选举的前提——先侦测到 leader 异常,才进入选举流程。
