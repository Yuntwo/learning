# 分布式事务实战方案汇总

> 来源：https://www.cnblogs.com/yizhiamumu/p/16625677.html
> 作者：一只阿木木
> 原文发布：2022-08-25
> 记录时间：2026-04-17

---

## 核心观点

分布式事务的复杂度和参与方数量成正比，没有银弹——应根据业务对一致性的要求（强/弱）选择方案，而不是追求统一的"完美事务"。互联网场景下，本地消息表 + MQ 的最终一致性方案最常用；只有资金类高价值业务才值得上 TCC。

---

## 要点整理

### 1. 理论基础：CAP 与 BASE

**CAP 定理**
- 共享数据系统最多同时满足 C（一致性）、A（可用性）、P（分区容忍性）中的两个
- 实际系统中 P 不可或缺（网络分区必然发生），所以真正的选择是在 CP 和 AP 之间
- 生产系统通常是 ACID 与 BASE 的混合体，而不是非此即彼

**BASE 理论**
- Basically Available（基本可用）：允许损失部分可用性
- Soft State（软状态）：允许中间状态存在
- Eventually Consistent（最终一致性）：一段时间后达到一致

一致性分三级：**强一致性 > 最终一致性 > 弱一致性**。最终一致性本质是弱一致性的一种受控形式。

---

### 2. 四种主流方案对比

| 方案 | 一致性 | 复杂度 | 适用场景 | 互联网常用 |
|---|---|---|---|---|
| XA（2PC） | 强 | 低 | 单体应用跨库 | 否（违反微服务原则） |
| TCC | 强 | 高 | 资金类、短事务 | 核心业务 |
| 本地消息表 | 最终 | 中 | 常规业务 | 是 |
| 可靠消息（MQ 事务消息） | 最终 | 中 | 解耦异步场景 | 是 |

---

### 3. XA 两阶段提交

核心流程：
- Phase 1（Prepare）：事务管理器向所有资源管理器发起预提交询问，各方锁定资源并回复 Yes/No
- Phase 2（Commit/Rollback）：根据回复结果统一提交或回滚

**问题**：
- 依赖数据库厂商实现（MySQL XA 性能差）
- 同步阻塞，资源锁定时间长
- 单点故障：协调者挂了所有参与者都卡死
- 违反微服务"服务自治"原则

---

### 4. TCC（Try-Confirm-Cancel）

三阶段含义：
- **Try**：检测 + 预留资源（而不是真的执行业务）
- **Confirm**：所有参与者 Try 成功后，执行真实业务
- **Cancel**：任何一方 Try 失败，所有已 Try 的参与者执行补偿

**例子**：账户扣款 100 元
- Try：冻结 100 元（available -= 100, frozen += 100）
- Confirm：扣掉冻结（frozen -= 100）
- Cancel：解冻（available += 100, frozen -= 100）

**痛点**：
- 每个接口要写三份代码（正向 + 确认 + 补偿）
- 业务侵入性强
- 幂等、空回滚、悬挂等边界问题都要自己处理

---

### 5. 本地消息表

核心思想：**业务数据和消息写入同一个本地事务**，保证原子性。

```
BEGIN;
  INSERT INTO order ...;                     -- 业务
  INSERT INTO local_message (status='new');  -- 消息
COMMIT;

-- 独立的 worker 轮询 local_message 表：
-- 1. 发送到 MQ / 调用下游
-- 2. 成功后标记 status='done'
-- 3. 失败则重试
```

**优点**：不引入额外中间件，简单可靠
**缺点**：
- 消息表数据量大时轮询压力大
- 高并发下性能瓶颈明显
- 要自己实现重试、幂等、死信等机制

---

### 6. 可靠消息最终一致性（RocketMQ 事务消息）

基于 MQ 自身事务能力，比本地消息表更优雅：

```
1. Producer → MQ：发送 half message（对消费者不可见）
2. MQ → Producer：ACK
3. Producer：执行本地事务
4. Producer → MQ：commit 或 rollback
   - commit：消息对消费者可见
   - rollback：消息被删除
5. 若步骤 4 超时：MQ 回查 Producer 的事务状态
```

相比本地消息表：
- 不需要额外消息表
- MQ 自己负责重试和回查
- 更适合微服务解耦场景

---

### 7. 实战场景对号入座

**强一致性场景**：
- 新老系统双写库同步 → 本地事务表 + 轮询补偿
- 分库分表路由字段绑定 → 本地事务表 + 事务消息
- 积分商品兑换 / 广告结算 / 抽奖活动 → TCC

**弱一致性场景**：
- 数据变更同步下游 → 最大努力通知（重试 N 次后丢弃）
- 秒杀库存回滚 → 战略放弃，允许极端情况下少量超卖/少卖

---

### 8. 几个必须处理的关键保证

**幂等防重**
- 业务 ID + 操作类型 组成唯一键
- 数据库唯一索引兜底（应用层判断 + DB 约束双保险）
- 去重表 / Redis SetNX

**尽早干预**
- 串行多步操作中，任一步失败立刻触发逆向补偿
- 不要等到最后统一回滚

**补偿驱动**
- 定时任务扫表 + MQ 重试是两种互补机制
- 定时任务兜底，MQ 扛主流量

---

## 实践印证

作为高并发交易结算系统后端工程师，这套方案在日常工作中对应：

- **结算流水** → 本地消息表：业务 + 消息同事务写入，保证账务和下游通知不丢
- **跨账户资金划转** → TCC：冻结-扣减-解冻三阶段，配合账务核对兜底
- **T+1 对账** → 最大努力通知 + 定时补偿：允许短暂不一致，通过日终对账最终收敛
- **幂等设计** → 业务流水号 + DB 唯一索引是底线，任何外部调用入口都必须有
