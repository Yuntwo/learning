MCP 和 RPC 到底差在哪？

[小红书正文]
很多人第一次接触 MCP，会把它理解成“Agent 之间通信的协议”。

这个说法方向没问题，但不够准。

更准确的理解是：MCP 本质上也是一种应用层协议，和 RPC 一样，都在解决“调用远端能力”这件事。

它们的相同点很明显：都有 client / server、method、params、result、error，也都可以跑在 stdio / HTTP 之上。

真正的区别，不在“能不能远程调用”，而在 MCP 额外标准化了面向模型的语义。

RPC 更关注的是：程序怎么调远端函数。
MCP 更关注的是：模型怎么发现工具、理解参数、决定是否调用、拿到结构化结果。

所以 MCP 不只是一个“远程调用协议”，它更像是给大模型和 Agent 用的能力接口标准。

你可以把它记成一句话：RPC 解决“怎么调”，MCP 解决“模型怎么稳定、结构化地调”。

如果你熟悉 LSP，这个类比也很好记：JSON-RPC 是通用通信壳子，LSP 是给编辑器和语言服务定义标准语义；MCP 则是在类似 RPC 的基础上，给模型和外部能力定义标准语义。

再落到工程实践里，像 Eino 里的 schema.ToolInfo，其实是框架内部的 tool 抽象，不是 MCP 原生协议对象；真正接到 MCP 时，通常还会有一层映射或桥接。

所以别把它们混成一层：
RPC：通用远程调用机制
MCP：面向模型的标准能力协议
Agent 框架：在上面做编排、选择和执行

一句话收尾：MCP 和 RPC 没有本质鸿沟，但 MCP 比 RPC 多了一层“给模型用”的标准语义。
