# Harness Engineering 实践：Lemon8 iOS 工程配置化改造

> 来源：https://bytetech.info/articles/7626270088611201034（ByteTech · 李宽）
> 记录时间：2026-04-21

---

## 核心观点

**AI 的任务完成度不取决于模型能力，而取决于它的"工作环境设计"。**
Harness Engineering 的本质是把工程师的经验、判断力、品味，从隐性知识转化为仓库中可执行的显性规则；人从"写代码"转向"搭环境"，AI 在环境里自验证、自修正直到收敛。本文用 Lemon8 iOS 工程从 13000 行 `project.pbxproj` 迁移到 380 行 `Lemon8.yml` 的真实案例，演示了怎样为 AI 搭一条"验证闭环"，把原本 2–3 周的人力压到等效 2 天。

## 要点整理

### 1. Harness Engineering 的四根支柱（OpenAI 框架）

OpenAI 三个工程师用 Codex 五个月造了一个百万行代码产品，1500 个 PR，零手写代码——他们把这套实践叫 Harness Engineering，"人类掌舵，智能体执行"：

- **环境可读性**：架构决策、设计规范、产品意图全部版本化进仓库。存在于 IM、经验、会议共识中的隐性知识对 Agent 等于不存在。
- **约束编码化**：工程规范写成 linter 规则、CI 检查、架构测试等**可执行**形式，而不是只靠文档约定。对人主导的团队可能过严，对 Agent 是能力倍增器。
- **反馈回路**：让 Agent 自动检查产出、发现问题、自修正、重新验证。人不逐行审代码，只审"验证机制的覆盖度"。
- **熵管理**：后台 Agent 定期扫描不合规模式、发起重构，类比垃圾回收——技术债是高息贷款，持续小额偿还好过一次性清理。

### 2. 三层 Harness 的划分

> **Agent = Model + Harness**

| 层级 | 谁建设 | 作用 |
| --- | --- | --- |
| **工具级** | Claude Code / Codex / Cursor 内置 | 文件读写、终端、搜索、Agent Loop——项目无关的通用底座 |
| **项目级** | Infra 团队 | Lint、核心自动化测试、CI/CD、关键节点 Hook——团队共享的硬约束 |
| **需求级** | 开发者自己 | 为特定任务临时拼装的工作环境，追求局部最优，即用即抛 |

短中期内需求级 Harness 落地效果最好，通用性让位于个性化；长期愿景是 Agent 自动编排。

### 3. 为什么 iOS 工程配置迁移天然适合 Harness

四个特征同时满足——这是能构建可验证闭环的**前提条件**，缺一不可：

- **规模大、细节多**（13000+ 行，几十个维度）→ 人力逐项不现实
- **正确性要求极高**（每个编译参数都要精确对齐，没有"大致可用"）
- **存量知识密集**（XcodeGen + 内部插件链）→ AI 消化源码比搜文档高效
- **可验证性强**（构建产物 + LinkMap + BUILD.bazel 可脚本对比，客观判定）

反过来，若结果依赖主观判断（UI 还原度）、验证靠人工（交互体验）、或单次迭代周期过长（必须部署上线才能验证），Harness 的收益显著降低。

### 4. 实战：四个落地组件

#### 多 Agent 架构（Planner + Generator + Evaluator）

Anthropic 博客的灵感来自 GAN——把"生成"和"评判"分离，解决单 Agent 的两大顽疾：
- **Context anxiety**：上下文将满时 Agent 会提前收尾
- **Self-evaluation bias**：Agent 自信地夸赞自己的输出

项目里的实际分工：
- Claude Code plan mode 制定迁移方案
- Claude Code 执行方案、自主循环迭代
- **Codex 审查改动**、生成审查报告交给人工

#### 反馈回路（三条自动 + 一条人工）

自动反馈：
- 构建报错 ← 插件体系
- 产物对比报错 ← **AI 自己写的对比脚本**（随任务迭代完善）
- 编译错误 ← jojo 工具链

人工反馈：每个关键阶段让 Codex 输出结构化审查报告，重点关注"无法直接迁移、靠脚本/硬编码辅助"的妥协项，以及 AI 删除的历史无用文件。

#### 会话交接（轻量而非重型记忆系统）

上下文将耗尽或输出漂移时，让 AI 写一份交接文档（进度 + 未完成事项 + 关键决策 + 注意事项）。新 Session 读文档即可继续，不同 Agent 之间也靠它共享信息。

**关键细节**：作者用两个 Skill（`handoff-l8` / `pickup-l8`），并特意设 `disable-model-invocation: true`——**交接时机由人判断**，因为长时间工作后模型质量已明显下降，不能让它自己决定何时交接。

#### 上线标准（本地闭环充分 → CI 一次过）

- 本地：备份原 `.xcodeproj` 和 `BUILD.bazel` 作为基线 → 对比脚本跑差异 → 修复到归零
- CI：适配路径、环境变量、签名等差异
- 产物对比：基于 CapCut JoJo Build 2.0 的对比脚本，重点比对 ipa / LinkMap / BUILD.bazel
- QA：两次完整回归

> CI 阶段一次性通过——**本地验证越充分，后续集成返工越少**。

### 5. 数据：验证闭环的收益量化

| 阶段 | 无闭环 | 有闭环 | 增益 |
| --- | --- | --- | --- |
| YML 编写 | AI 出首版覆盖 ~80%，剩 20% 人工逐项核对 | AI 自动对比、自动修正直到归零 | **效率 5×，完成度逼近 100%** |
| 文件清理 | 人工排查易遗漏 | 自动扫描 + 构建验证 | 零遗漏 |

> 缺少闭环时，AI 首版通常只到 80%——最后 20% 需要人兜底。有闭环后，人的重心从"这行代码对不对"转向"验证覆盖够不够"。

### 6. 实操建议（原文四条）

1. **优先构建反馈回路**——即使验证环境搭建成本和任务本身相当，复利回报仍可观
2. **知识沉淀进仓库**——Agent 触达不到的信息等于不存在，信息可访问性决定 Agent 工作质量上限
3. **前移 Code Review 重心**——从逐行审代码 → 评估验证策略和约束规则的覆盖度
4. **常态化清理**——AI 产出越多，代码漂移越严重，质量巡检要像 GC 一样持续运行

## 实践印证（本项目可借鉴处）

支付 RPC 项目里能对齐的点：
- **约束编码化**：现有 MR 模板、lint、CI 都属于项目级 Harness；扩展方向是把支付侧的一致性检查（idempotent key、重试幂等、金额精度）写成可执行的 lint 规则，而不是 code review 里口头提。
- **反馈回路**：收银台/结算侧改动多属"可客观判定"的类型（接口契约、对账一致性），天然适合让 AI 写校验脚本并迭代。
- **熵管理**：`harmony`、`feat: 清理diff 代码`这类清理 commit 现在是零散的，按文章建议应该常态化。
