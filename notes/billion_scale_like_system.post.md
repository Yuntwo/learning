## 小红书

**标题**：千亿级点赞系统拆解

**正文**：
```text
点赞系统看起来只是一次“+1”，但真正上生产后，难点全在高并发和热点治理。

这次整理一篇大型内容社区的点赞架构，最有价值的是 5 个点：

1️⃣ 记录和计数一定要拆开
点赞记录负责“谁在什么时候点了什么”，点赞计数负责“这个内容现在有多少赞”。两类数据的查询方式完全不同，混在一起会很痛苦。

2️⃣ 三层存储不是炫技，是刚需
持久化账本放 DB，高频读写放 Redis，超级热点再下沉到 LocalCache，本质是把不同压力分散到最合适的介质上。

3️⃣ 热点问题必须单独治理
热门内容会把流量打到单分片、单 key。只靠普通缓存不够，还要做热 key 识别、本地缓存、TTL 控制，必要时继续分桶。

4️⃣ 用户链路快返回，重操作异步化
点赞、刷新缓存、写持久层、发消息要尽量拆开。能异步的尽量异步，能聚合写的尽量聚合写，主链路只保用户体验。

5️⃣ 故障时优先“用户无感”
多机房、多副本、多存储互相兜底；哪怕全链路异常，也尽量返回默认态，等恢复后再补写和对账。

我还顺手补了一个 Redis + Lua 的简化版点赞 demo，把“点赞状态 + 点赞计数 + 用户点赞列表”放进一个原子脚本里，特别适合理解这种系统的最小闭环。
```
