[小红书标题]
Claude Code 的多轮上下文是怎么带的？

[小红书正文]
如果你抓过一次 Claude Code 的 `v1/messages` 请求，通常都会有一个第一反应：
怎么里面不只是当前这句话，而是已经带上了好多前文，甚至还有工具调用和工具结果？

我最近把这件事专门拆了一下，发现它其实可以用一个很清楚的模型来理解：

Claude Code 的多轮上下文，不是服务端偷偷帮你记住了一切，而更像是客户端或 harness 在每次发新请求时，把“当前系统注入 + 多轮历史 + 工具轨迹 + 必要摘要”重新组织后，再一起发给模型。

先说第一个最重要的区分：
system 和 messages 不是一回事。

system 更像全局说明层。
这里通常会放：
- 你是谁，比如 Claude Code
- 当前环境信息
- skills 列表
- memory 规则
- harness 约束

如果你在抓包里看到：
The following skills are available for use with the Skill tool:
那大概率就是运行时注入进来的系统级上下文，而不是普通聊天历史。

messages 才是多轮会话历史层。
这里会放：
- 用户之前说了什么
- assistant 之前回了什么
- 工具调用轨迹
- 工具结果回传
- 有时还包括 slash command 注入信息

所以，当你看到一个新请求里已经带着前面好几轮问答，这不是 API 自动给你补出来的，而是客户端主动把它们一起重发了。

这里最容易误解的一个点是：
是不是所有东西都被放进了 `role=user`？

不是。

更准确的结构是：
- 用户自然语言输入，在 `role=user`
- 模型自然语言输出，在 `role=assistant`
- assistant 发起工具调用时，会出现 `tool_use`
- 工具执行完之后，结果常常会作为下一条 `role=user` 里的 `tool_result` 回传给模型

这也是为什么很多人第一次看抓包会觉得怪：
怎么工具结果也在 user 那边？

因为从协议角度看，那一步不是“模型自己生成了结果”，而是“外部工具把结果喂回模型”。
所以它虽然挂在 user 一侧，但语义上不是普通用户发言，而是系统代替外部世界给模型补充输入。

如果用一个最小示意来理解，大概是这样：

第一步，用户提问。
第二步，assistant 发起工具调用。
第三步，工具返回结果。
第四步，assistant 基于结果继续回答。

你可以把它想成一个 agent loop，而不是普通聊天框的一来一回。

再说第二个很关键的问题：
如果会话特别长，会不会一直原样把所有历史都放进去？

通常不是。

Claude Code 在长会话里，很可能会做 context management，也就是：
- 保留最近若干轮原始内容
- 把更早的对话压缩成摘要
- 用“摘要 + 最近原文”的方式继续往后传

这也解释了两个现象：

现象一：短会话抓包时，你能看到很多前文原样还在。
现象二：会话越来越长之后，最早那些轮次不一定还逐字保留，而可能变成一段 summarize。

所以更准确的说法不是“Claude Code 永远原样重放整个聊天历史”，而是：

它会尽量保留必要上下文；如果上下文压力变大，就把旧内容压缩成更短的可继续推理的表示。

如果把整个机制压成一个最小 mental model，我会这样记：

第一层：system
放规则、环境、skills、memory。

第二层：messages
放多轮 user / assistant 历史，以及 tool_use / tool_result。

第三层：context management
上下文太长时，把旧历史 summarize，再和最近原文一起传下去。

如果只记住一句话，我会这么总结：

Claude Code 的“多轮记忆”本质上不是神秘会话魔法，而是客户端持续重建上下文；而当上下文太长时，它会开始用 summarize 替代更早的原始细节。
