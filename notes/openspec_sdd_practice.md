# OpenSpec：把 AI 编程从“会写代码”推进到“按规范交付”

> 来源：https://developer.aliyun.com/article/1705378
> 记录时间：2026-07-02

---

## 核心观点

OpenSpec 的价值不在于替代编码，而在于先把需求、边界、任务和变更记录整理成机器可读的规范，再让 AI 沿着明确约束执行，从而降低“需求跑偏、实现失真、返工增多”这三类常见问题。

---

## 要点整理

### 1. 为什么 AI 辅助开发容易失控

文章先指出一个很现实的问题：只靠自然语言提示驱动 AI 写代码，单次看起来很快，但项目级别的不确定性会显著上升。

常见表现包括：

- 需求细节理解不完整，结果和预期有偏差
- AI 会补出“看似合理但其实没人要”的功能
- 待办、边界条件、兼容约束容易被漏掉
- 代码能生成出来，但和真实工程流程脱节

本质上，问题不是 AI 不会写，而是缺少一份足够稳定的“执行合同”。

### 2. OpenSpec 的核心思路：先写规范，再让 AI 执行

OpenSpec 属于规格驱动开发（Spec-Driven Development）的一种轻量实践。

它强调：

- 在编码前先定义变更目的与范围
- 把需求行为、边界和验收条件写成结构化规范
- 让规范成为人和 AI 共用的单一事实源
- 通过提案、审查、实施、归档的闭环管理变更

可以把它理解成“面向 AI 编程的 GitOps”：先声明，再执行；先对齐，再改动。

### 3. OpenSpec 适合做什么

从文章给出的对比看，OpenSpec 更适合已有项目中的持续演进，而不只是新项目初始化。

它比较适合这几类场景：

- 老系统新增能力，需要避免影响存量逻辑
- 多人协作，希望把需求和变更记录集中管理
- AI 参与较深，希望减少上下文漂移
- 需求比较复杂，必须先澄清边界才能安全实施

换句话说，OpenSpec 更像“变更治理层”，而不只是一个脚手架工具。

### 4. 它的工作流为什么有用

文章里的标准流程很清晰：

1. 初始化仓库规范结构
2. 为一个变更起草 proposal
3. 补充 spec delta 与任务清单
4. 人工审查 proposal / spec / tasks
5. AI 按已批准任务实施
6. 变更完成后归档回主规范库

这个流程的价值在于：

- **提案阶段** 解决“为什么做、做到哪”
- **规范增量阶段** 解决“行为应当是什么”
- **任务阶段** 解决“按什么顺序落地”
- **归档阶段** 解决“以后如何回看和继续演进”

如果没有这四层，AI 很容易直接跳到“开始写代码”，最后把原本应该在前面说清楚的决策推迟到实现阶段处理。

### 5. 目录结构体现了“现状”和“变更”的分离

OpenSpec 推荐把仓库内容拆成两类：

- `openspec/specs/`：系统当前已经成立的规范
- `openspec/changes/`：正在讨论或实施中的变更

每个变更一般包含：

- `proposal.md`：这次为什么改、改什么
- `tasks.md`：实施步骤清单
- `specs/.../spec.md`：本次变更对应的规范增量

这个设计很关键，因为它把“当前事实”和“待落地变化”分开了。AI 在执行时拿到的是更稳定的上下文，而不是在一堆聊天记录里反复猜测哪个结论才是最新版本。

### 6. 文中案例：用 OpenSpec 起草用户管理能力

文章用一个演出排班管理系统做示例，先补全项目上下文，再围绕“管理员创建和编辑用户”起草变更。

案例里比较值得借鉴的是：

- 先把项目技术栈、角色、认证、测试方式说明清楚
- 再定义用户管理能力的边界
- 然后拆到角色权限、用户 CRUD、音乐人资料、审计日志等子点
- 最后用校验命令检查 proposal / spec / tasks 的结构正确性

这说明 OpenSpec 不是只写一份大而空的设计文档，而是把“项目背景—功能提案—行为规范—实施清单”串成一个连续链路。

### 7. 学习 OpenSpec 时真正要掌握的三件事

比起记命令，更重要的是理解它背后的三层方法：

#### 第一层：规范怎么写

重点不是把文档写长，而是把下面几件事写清楚：

- 目标用户是谁
- 需要新增或改变什么行为
- 边界条件和异常情况是什么
- 与现有能力如何兼容
- 这次变更不处理什么

#### 第二层：变更怎么审

proposal 和 spec delta 需要在编码前先过一遍人审。

这一步能把很多“实现时才发现的问题”前移，比如：

- 范围是不是太大
- 是否遗漏兼容路径
- 角色或权限模型是否矛盾
- 任务拆分是否合理

#### 第三层：规范如何持续沉淀

如果每次改动结束都把结果归档回主规范库，长期来看就会形成一套不断更新的工程事实源。后续不管是人继续开发，还是 AI 接手，都能更快进入上下文。

## 实操命令速记

```bash
npm install -g @fission-ai/openspec@latest
openspec init
openspec list
openspec view
openspec show <change>
openspec validate <change>
openspec archive <change> --yes
openspec update
```

在支持的 AI 工具中，文章还给出了类似下面的交互入口：

```text
/openspec:proposal
/openspec:apply <change>
/openspec:archive <change>
```

这些命令本身不复杂，关键是它们对应的是一套“先规范、后执行、可归档”的开发闭环。

## 最适合引入 OpenSpec 的时机

如果团队已经出现下面这些信号，就很适合尝试：

- AI 经常把需求理解偏
- 同一个需求在不同轮次里描述不断漂移
- 代码评审时才发现边界遗漏
- 老项目加功能时，隐含约束全靠口口相传
- 变更记录分散在聊天、PR、会议纪要里，难以回溯

这时引入 OpenSpec，收益往往不只是“AI 更稳”，也是团队知识沉淀方式更稳。

## 实践印证

这篇文章和当前项目里的一个典型问题很契合：**登录态/uid 解析路径演进**。

在本仓库里，`utils/session.go:28` 到 `utils/session.go:108` 把 uid 解析拆成了多级来源：

- 先走网关 session 中的 `UserID`
- 再尝试历史 header `x-tma-host-sessionid`
- 再走 IDE session 兜底
- 同时通过 `metrics.EmitAuthUIDSource` 记录命中来源

对应的指标枚举定义在 `metrics/metrics.go:15` 到 `metrics/metrics.go:25`，已经把 `gateway_session`、`legacy_header_sessionid`、`ide_session`、`unresolved` 等来源显式建模了。

而在查询主链路里，`service/read/user_order_query.go:54` 到 `service/read/user_order_query.go:62` 又体现出一个更推荐的调用约束：HTTP 请求优先使用 request loader 注入的 `user_id`，再回退到上下文解析。

这正是 OpenSpec 适合发挥作用的地方。类似变更如果先写成规范，至少可以在编码前明确：

- 首选 uid 来源是什么
- 哪些链路允许回退，哪些不允许
- 历史 header 兜底是临时兼容还是长期能力
- 指标要如何区分 hit / miss / error
- 调用方应该读 loader 字段还是自己解析 session

也就是说，这类“认证来源迁移 + 兼容兜底 + 可观测性补齐”的需求，如果没有规范，很容易在多个 handler、工具函数和指标口径里逐渐分叉；而有了 spec，就更容易把行为、回退顺序和下线路径一次写清楚。

## 一句话收获

OpenSpec 真正解决的不是“AI 能不能写代码”，而是“AI 是否能在长期项目里沿着稳定约束交付正确变更”。
