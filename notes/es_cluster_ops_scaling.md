# ES 集群运维:架构关系、分片、主从备份与扩缩容决策

> 来源:钱包后台搜索集群(`byte.es.wallet_admin_search`)一次"逻辑卷分区使用率 > 85%"告警的排查过程
> 记录时间:2026-06-23

---

## 核心观点

ES 集群的**磁盘是按"节点/逻辑卷"为单位统计的,数据却是按 shard 分布的**。所以磁盘告警的本质永远是"哪些 shard 压在了哪些节点上",而不是卷本身的问题。看到磁盘水位倾斜(部分节点 85%、部分节点 13%)时,第一反应应该是**治分片倾斜**,而不是无脑扩容——扩容只会把同样的倾斜搬到更大的卷上。

---

## 要点整理

### 1. 架构关系(从上到下的五层映射)

```
ES 逻辑集群 (cluster, 一个 PSM)
   │
   ├── 节点池 / 分层 (node pool / tier)        ← 如 data、data2 两组
   │      └── 每组 N 个数据节点 (data node)
   │
   ├── 数据节点 = K8s 上一个 Pod (= 一个 ES 实例)
   │      └── Pod 挂载 PVC → 绑定 PV → 落到宿主机一块逻辑卷 (LVM)
   │             └── "逻辑卷分区使用率" = 这块卷上 data 目录的占用
   │
   └── 索引 (index)
          └── 切成 N 个主分片 (primary shard) + 副本 (replica)
                 └── 每个 shard 被分配到某个数据节点
                        └── 落到该节点的逻辑卷上:
                            /data/nodes/0/indices/<index_uuid>/<shard_no>/index/*
```

关键术语对应:
- **cluster(集群)**:一个逻辑 ES 服务,内部一个 PSM 标识。
- **node(节点) = 实例 = Pod**:在 K8s 化的 ES 上,一个数据节点就是一个 Pod,一个 ES 进程实例。
- **node pool / tier(节点池/分层)**:同一集群里按角色/批次分的节点组(如 `data` / `data2`,可能是 hot/warm 分层,也可能是扩容时新加的一批节点)。
- **逻辑卷(LVM volume)**:节点数据目录所在的磁盘分区,扩磁盘就是扩它(PVC expand)。

### 2. 索引 ↔ 逻辑卷:没有直接对应,是"多对多",中间隔着 shard

- **一块卷 : N 索引** —— 一块卷 = 一个节点的数据目录,上面躺着所有被分配到这个节点的 shard,这些 shard 来自几十上百个不同索引。卷使用率 = 这些 shard 大小之和。
- **一个索引 : N 卷** —— 一个索引 `主分片数 × (1 + 副本数)` 个分片,被打散到不同节点 → 落到不同卷。例如 6 主 1 副 = 12 个 shard,理想落在 12 个节点的 12 块卷上。

所以"逻辑卷 ↔ 索引"是经 **shard → node → 卷** 三跳间接关联的。**运维/扩容按"卷(节点)"为单位,治理倾斜按"shard/索引"为单位——这个错位是所有 ES 磁盘问题的根源认知。**

### 3. 主分片与副本(主从备份)

- **主分片(primary)**:数据的权威副本,写入先落主分片。索引创建时主分片数固定,**不能在线改**(要改只能 reindex / split / shrink)。
- **副本(replica)**:主分片的拷贝,提供高可用 + 分摊读。副本数**可以在线改** (`number_of_replicas`)。
- ES 保证**主分片和它的副本绝不分配到同一个节点**(否则节点挂了主副本一起丢),所以:
  - 副本会显著放大磁盘占用:1 副本 = 数据 ×2 落盘。
  - 节点数必须 > 副本数,否则会有副本分不出去(unassigned)。
- **临时止血技巧**:磁盘吃紧时可临时把副本数降到 0,立刻释放一半空间;事后再加回——但降副本期间没有冗余,有风险。

### 4. 磁盘水位线(watermark)——为什么 85% 是该动手的点位

ES 有三道默认水位线,触发后行为不同:
| 水位线 | 默认值 | 行为 |
|--------|--------|------|
| low | 85% | 不再把新 shard 分配到该节点 |
| high | 90% | 开始把该节点上的 shard 往外迁 |
| flood_stage | 95% | 给该节点上的索引加 `read_only_allow_delete` 只读锁 |

所以告警卡在 85% 是合理的预警点:再往上到 90% 会触发自动搬迁(抖动),到 95% 索引会变只读(写入直接失败)。

### 5. 倾斜 vs 容量不足:先做的判断

把节点按磁盘使用率排序:
- **整体都贴近 85%、且分布均匀** → 真·容量不足 → 该扩容。
- **最高 85%、最低却只有 13~40%(差几倍)** → 这是**分片倾斜**,不是容量问题 → 先 rebalance / 治倾斜,扩容是次要。

倾斜常见原因:
- 大索引 shard 数太少(如几百 GB 却只有 2~3 个 shard),死死压在两三块卷上,**加节点也摊不开**,只能 reindex 提高 shard 数。
- 个别大 shard 恰好被分到同一批节点。
- 多节点池角色不同,只有一组在被重写入。

### 6. 两种扩缩容方式的选择

| | 水平扩容(加节点) | 节点卷/磁盘扩容(扩 PVC) |
|---|---|---|
| 做什么 | 增加 data 节点 Pod,ES 自动 rebalance shard 到新节点 | 扩大每个节点 PVC/逻辑卷容量 |
| 同时提升 | 磁盘 + 查询/写入算力 + 故障域 | 仅磁盘 |
| 速度 | 慢(要搬数据、rebalance) | 快(storageClass 支持 expand 时几乎无搬迁) |
| 适用 | 数据持续增长、查询压力也涨、shard 数足够能摊开 | 数据已均衡、纯缺盘、要快速止血 |
| 限制 | shard 太少/太大时加节点摊不动 | 受宿主机单盘容量 & 单 Pod 配额上限 |

**缩容**同理反向:
- 水平缩容(下节点):先 `_cluster/reroute` / 设 `exclude._name` 把待下线节点的 shard 排空,确认 0 shard 后再摘 Pod,否则丢数据。
- 磁盘缩容:LVM/PVC 一般**不支持在线缩小**,通常要新建小卷 + 迁移,成本高,实际很少做。

### 7. 处理顺序(从轻到重,务必按这个顺序)

1. **止血**:调 `cluster.routing.allocation.*` / 手动 reroute,把高水位节点的 shard 迁走;必要时临时降副本。
2. **查根因**:`_cat/allocation` 看每节点磁盘+shard 数;`_cat/shards` 按 `store` 排序,过滤高水位节点,定位是哪个索引的大 shard。
3. **清存量**:缩短保留期 / rollover 删旧索引 / force merge / 降副本。
4. **再谈扩容**:均衡缺盘→水平扩节点(ES 首选);纯救急不想搬数据→扩 PVC。

## 止血手段详解:删数据 vs 降副本 vs 扩磁盘(原理与取舍)

> 补充于 2026-06-23:同一集群第二次告警(`data-8` 85.02%)时整理。核心是搞清"为什么降副本能秒级省一半、且暂时不影响使用",以及它的代价。

### A. 降副本(`number_of_replicas: 0`)—— 秒级释放一半,但换掉了高可用

**为什么省一半:** 副本是主分片的**完整拷贝**,`1 副本` = 每份数据在磁盘上存 2 遍(主+副)。副本数 1→0,ES 立刻删掉所有副本分片,直接释放"第二遍"的空间。

**为什么"暂时无影响":** 读写都走**主分片**,副本没了主分片还在 —— 读能查、写正常、数据正确性不受影响。

**代价(这是"暂时"二字的重量):**
1. **集群 green→yellow,进入无冗余窗口** —— 此时**任一持有主分片的节点宕机 = 该分片数据丢/不可用**(除非有快照恢复)。
2. **读容量下降** —— 副本本也分摊读,降了之后读全压主分片,高 QPS 下会抖。
3. **加回要重建** —— 事后恢复 `number_of_replicas: 1`,ES 要把主分片重新拷一遍,磁盘涨回去 + 期间 IO/网络负载。

**适用:** 数据有上游可重建、能接受短时无 HA(如后台搜索类)→ 拿来争取时间最划算。**切忌长期挂 0 副本。**

```
PUT <index>/_settings { "number_of_replicas": 0 }
```

### B. 删数据(删索引 / 缩短保留期)—— 真·永久减容

- `DELETE <index>`:整个索引(主+副)从磁盘抹掉,空间立即回收。时序数据用 rollover + 删最老索引。
- **关键坑:按文档删(`delete_by_query`)不立即释放磁盘!** 只打"删除墓碑(tombstone)",空间要等**段合并 / force_merge** 才回收:
  ```
  POST <index>/_forcemerge?only_expunge_deletes=true   # IO 重,低峰执行
  ```
- 所以"删数据腾盘":能删整索引就 `DELETE`(立竿见影);只能删部分文档则删完还得 force_merge。

### C. 扩磁盘(PVC expand)—— 不动数据、不丢冗余

保副本、保数据,纯加卷。最快最安全,但只解决"盘不够",不解决倾斜也不补算力。storageClass 支持在线扩容时几乎无搬迁。

### 选择速查

| 场景 | 首选手段 |
|------|---------|
| 数据旧/可删(日志、过期) | **删索引**(永久,最优) |
| 数据要留、急救、能忍短时无 HA | **降副本**(秒级省一半,可逆,有风险窗口) |
| 数据全留、要保 HA、纯缺盘 | **扩 PVC**(安全,不动数据) |
| 倾斜(部分 85% 部分 13%) | 先 **rebalance**,以上都是治标 |

> ⚠️ 降副本/扩磁盘是作用到该索引**所有卷**的节点级操作;若本质是倾斜,降副本只是把倾斜整体缩小一半,倾斜本身还在。

## 实操命令(via Kibana Console / _cat API)

```bash
# 每个节点(=每块卷)的磁盘占用 + shard 数,按磁盘降序
GET _cat/allocation?v&s=disk.percent:desc

# 哪些 index 的哪些 shard 落在哪个节点,按 shard 大小降序
GET _cat/shards?v&h=index,shard,prirep,state,store,node&s=store:desc
# 再过滤到高水位节点名,store 最大的几个 index 就是把卷撑高的元凶

# 看未分配分片及原因(扩缩容、降副本后常用)
GET _cat/shards?v&h=index,shard,prirep,state,unassigned.reason | grep UNASSIGNED

# 在线改副本数(止血 / 恢复)
PUT <index>/_settings { "number_of_replicas": 0 }
```

## 踩坑记录:用 bytedcli 连内场 ES

- 命令:`bytedcli es search --psm <psm> --idc hl --index "_cat/allocation?v" --query '{}' --method GET`
  - `es search` 表面是"index + DSL",但 `--index` 传 `_cat/xxx` 路径 + `--method GET` 可以走 `_cat` API。
- **doas 报 `DOAS_INSTALL_PATH_ERROR`**:bytedcli 把 `doas` 二进制丢在 cwd 但 cwd 不在 PATH。解决:把 `doas` 拷到 `~/.local/bin`(已在 PATH)。
- **doas 区域检测错误**:自动按本机 IP 检测区域失败时会默认 `boe`(测试环境),查 prod cn 集群会查错地方。必须显式 `DOAS_ZONE=cn`(doas 的 `-e/--env` 取值:boe/cn/row/nontt/euttp/usttp)。
- **首次需扫码**:doas 取 GDPR token 走 SSO 扫码(飞书扫码),会阻塞等待;扫一次后 token 缓存在 `/tmp/doas-[uid]-[user]`,后续命令复用不再弹码。非交互环境用 `--no-interaction` 只会打印提示并退出。

## 一句话记忆

> **卷是统计单位,shard 是数据单位,index 是治理单位。磁盘告警先看倾斜还是真满:倾斜先 rebalance,真满再选扩容——加节点补算力+磁盘+故障域,扩 PVC 只补磁盘但最快。副本数能在线改、主分片数不能(要 reindex);降副本是止血神器但丢冗余。**
