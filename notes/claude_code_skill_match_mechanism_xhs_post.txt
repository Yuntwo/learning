Claude Code 的 skill 是怎么命中的？

[小红书正文]
最近看了一次 Claude Code 的 `v1/messages` 抓包，我对它的 skill 命中机制有了一个更清晰的理解。

一个很关键的观察是：它并不像很多人直觉里那样，靠某个“隐藏分发器”在服务端偷偷路由。至少从这次请求看，更像是客户端或运行时先把当前可用的 skills 整理成一份能力清单，再把这些 skill 的名字、用途说明、触发词一起塞进 system 上下文里，让模型自己在 prompt 内判断该提到谁、该调用谁。

为什么这个观察值得看？
因为它把“skill 命中”这件事从黑盒，拆成了几个很清楚的层次。

第一层，是发现。
运行时先知道当前有哪些 skills 可用。

第二层，是识别。
模型读到用户问题后，会结合 skills 列表里的 description 和触发词，判断哪个 skill 跟问题最相关。

第三层，是执行。
只有当模型真的发起 `Skill` 工具调用时，才算正式进入 skill 的执行流程。

这三层不要混在一起。
很多时候，模型只是“知道有这个 skill”，不代表它“已经调用了这个 skill”。

我这次看到的最强证据，是请求里直接出现了一段很明确的 system 内容，大意就是：

The following skills are available for use with the Skill tool:

后面紧跟着整份 skills 列表。
其中不但有 skill 名，还带了说明、触发词、适用场景。

这意味着什么？
意味着模型不是临时去本地目录里找 skill 文件，而是直接读取了一份已经注入上下文的“技能注册表”。

如果把它和传统插件路由思路做个对照，会更容易理解：

对照项              传统直觉理解                这次抓包更像看到的方式
skill 在哪被发现     服务端内部逻辑              运行时预先整理后注入上下文
怎么判断该用谁      隐式规则或黑盒路由          模型读 description/触发词后判断
tool 何时执行        一识别就立刻执行            识别和执行是两步
可解释性            较弱                        很强，抓包里能直接看到依据

从工程角度看，这种设计其实很合理。

第一，它很统一。
只要一个 skill 能注册进来，就能通过同一套上下文注入机制暴露给模型。

第二，它很可扩展。
新增 skill 的重点不只是“写出功能”，而是把 description、触发词、边界写清楚。因为这些文本本身就是模型做决策的依据。

第三，它很可解释。
当你发现模型提到了某个 skill，往往可以直接回到 system prompt 里，看到它为什么会提到这个 skill。

不过也要注意边界。
一次抓包，能证明的东西其实有限。

比如下面这些问题，单靠一次 `v1/messages` 还回答不了：

1. skills 列表是不是每次请求都重新扫描
2. skills 是来自本地目录、缓存，还是多个来源合并
3. 注入前是否做了排序、裁剪或预匹配
4. slash command 和自然语言触发，是否走了不同链路

所以更稳妥的表述应该是：

从这次抓包强烈可以推断，Claude Code 的 skill 命中机制，第一步非常像是“运行时先把 skill 注册表注入 system 上下文”，然后由模型在 prompt 内完成识别和选择。

如果只记住一句话，我会这么总结：

skill 体系的关键，不只是有没有工具，而是有没有一份写得足够清楚的能力描述，让模型可以把它当作路由协议来理解。
