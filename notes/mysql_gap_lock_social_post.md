# 社交发布预览：MySQL 间隙锁

## X / Twitter

很多人以为 `for update` 只会锁住“查到的那一行”，但在 InnoDB RR 下，经常锁住的是一整个索引区间。

我把“间隙锁”重新过了一遍，最后压成 6 个最容易踩坑的判断：

1. 唯一索引等值命中，通常退化成记录锁
2. 查不存在的唯一键，也可能锁住相邻 gap
3. 唯一索引范围查，会锁到第一个不满足条件的值
4. 非唯一索引范围查，锁范围通常比直觉更大
5. gap lock 之间不一定互斥，但插入意向很容易把事务拖进死锁
6. `limit` 会改变扫描终点，也会改变锁范围

拿项目里的 `aweme_wallet_content_trade_db.content_order` 套进去看更直观：

- `order_id + order_type` 这种唯一索引锁单，影响面最小
- 用户维度批量 `for update`，更容易把相邻插单一起卡住
- “业务唯一”不等于“数据库唯一”，没唯一索引就别按单行锁理解

最值得记的一句：
不是“你写了什么 where”，而是“引擎为了满足这个 where，实际扫描到了哪里”。

## 小红书

### 标题

搞懂 MySQL 间隙锁的 6 个场景

### 正文

最近重新梳理了一遍 MySQL 的间隙锁，最大的感受是：

很多线上锁等待，不是因为锁住了某一行，而是锁住了“索引之间的空档”。

把概念压缩一下：

- 记录锁：锁住已经存在的那一行
- 间隙锁：锁住索引值之间的空档，重点是防插入
- 临键锁：记录锁 + 间隙锁，默认按这个思路理解最稳

最容易踩坑的 6 个场景：

1. 唯一索引等值命中  
如果记录存在，通常退化成记录锁，只锁那一行。

2. 唯一索引等值查不存在  
虽然没查到数据，但可能把相邻 gap 锁住，所以别的事务插入会被卡。

3. 唯一索引范围查  
会锁到第一个不满足条件的值，不要按“只查到 1 行就只锁 1 行”理解。

4. 非唯一索引范围查  
这是最容易低估锁范围的场景，锁通常会比你想的更大。

5. 间隙锁死锁  
两个事务都先拿到 gap 范围，后面都想往里面插数据，就可能互相等死。

6. `limit` 对加锁有影响  
因为引擎只锁扫描到的对象，所以 `limit` 会改变扫描停止点，也会改变锁范围。

我还把这套逻辑映射到了业务库 `aweme_wallet_content_trade_db.content_order`：

- `order_id + order_type` 这种唯一索引，更适合精确锁单
- 按 `user_id/status` 批量扫订单做 `for update`，更容易把同范围插单一起卡住
- `biz_order_id` 如果不是唯一索引，就不要按“只锁一条”来想

一句话总结：

判断锁范围时，不要只看 SQL 长什么样，要看它走了什么索引、扫到了哪里、在什么位置停下来。
