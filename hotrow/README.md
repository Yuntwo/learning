# 单行热点更新:TPS / RT 的真相与常见误解

> 起因:读腾讯云 MariaDB《利用热点更新技术应对秒杀场景》一文,搞不清"同一行串行更新"
> 到底意味着什么。本目录用 `main.go`(`sync.Mutex` 模拟单行行锁)实测,沉淀结论。
>
> 原文链接: https://www.tencentcloud.com/zh/document/product/237/13402

---

## 一、那篇文章到底在讲什么

秒杀/抢购时,同一个商品库存在 DB 里是**一行**。高并发下大量线程争抢这一行的
**InnoDB 行锁**,排队 → "并发越高,TPS 下降、RT 上升"。

文章介绍的"热点更新技术"是在 SQL 里加新关键字,把
**加锁 → 更新 → 判断成功/失败 → 提交/回滚** 压进数据库内核一次做完,
把锁占用时间压到最小,从而扛住洪峰:

| 关键字 | 含义 |
|--------|------|
| `COMMIT_ON_SUCCESS`  | 成功就立刻提交,尽早释放锁 |
| `ROLLBACK_ON_FAIL`   | 失败就立刻回滚,别占着锁 |
| `QUEUE_ON_PK expr`   | 指明热点对象(哪一行),为它建高效排队通道 |
| `TARGET_AFFECT_ROW n`| 校验影响行数(通常为 1),否则视为失败 |

⚠️ 它优化的是**锁占用时间(把 RT 压小 = 把天花板抬高一点)**,
**不能**让一行被多个事务同时写,**突破不了单行的吞吐天花板**。

## 二、四个最容易误解的点(逐个纠正)

### 误解 1:"同一行只能一个一个更新,所以 TPS = 1"

❌ 错。**串行 = 排队一个接一个,但每个很快**。

```
TPS(单行) ≈ 1 / 每个事务的处理耗时(RT)
```

demo 里持锁 50µs,则 `TPS ≈ 1/50µs = 20000`,不是 1。
"一次只能一个" ≠ "一秒只能一个",而是"一秒能塞进很多个"。

### 误解 2:"提高并发数基本一定能提高 TPS"

❌ 只在**瓶颈有空闲**时成立。瓶颈一旦打满,加并发 TPS 再也提不动。

TPS 随并发是一条三段曲线:

```
TPS
 │           ┌──────────────  ② 饱和/③过载:TPS 封顶(③甚至反降),加并发无效
 │          /
 │         / ① 欠载:瓶颈有空闲,加并发 → TPS 近似线性涨(这时"加并发提TPS"才对)
 │        /
 └──────────────────────→ 并发数
        ↑ 拐点 knee:瓶颈饱和
```

| 阶段 | 瓶颈 | 加并发效果 | RT |
|------|------|-----------|-----|
| ① 欠载 | 有空闲 | TPS 涨 | 基本不变(≈服务时间) |
| ② 饱和 | 刚满 | TPS 封顶 | 开始涨 |
| ③ 过载 | 满+内耗 | TPS 可能**反降** | 线性暴涨 |

本 demo 里每个 goroutine 进去只抢这把锁,**并发=1 时锁就已 100% 忙**,
所以从一开始就在阶段②/③ → 实测 TPS 全程封顶 ~19700,加并发完全不涨。

### 误解 3:"RT 那么大,相同 duration 里跑不完,完成数应该更少"

❌ 漏了"同时在跑的 goroutine 数"。正确公式:

```
完成数 = 并发数 × (duration ÷ 每个goroutine的RT)
```

- 并发 32: `2s/1.6ms ≈ 1250` 笔/工人 × 32 = 40000
- 并发 256:`2s/13ms ≈ 154`  笔/工人 × 256 = 39400

每个工人确实跑得少(154 < 1250),但工人多了 8 倍,**正好抵消**。
因为 RT 随并发线性涨,公式里并发被约掉:

```
完成数 = 并发 × duration / (并发 × 50µs) = duration / 50µs = 恒定
```

物理直觉:RT=13ms 里只有 50µs 在真正干活,~12.95ms 全在**排队等**,
而排队是一大堆人同时排、不占瓶颈资源的。**吞吐只由瓶颈出货速度(50µs/笔)决定。**

### 误解 4:"RT 升高 = 处理性能本身下降了"

⚠️ 不准确。RT 涨的部分**几乎全是排队等待**,不是处理变慢:

```
RT(13ms) = 排队等待(~12.95ms,在膨胀) + 真正干活(50µs,恒定)
```

服务能力没退化,是请求在**堆积**。只有极端过载(疯狂上下文切换、cache 抖动)
服务时间本身才会真的劣化——那是阶段③更深处。

## 三、核心规律(记住这几条)

1. **TPS = 系统整体出货速度,由瓶颈资源决定。**
   瓶颈没满 → 加并发能提;瓶颈满了 → 封顶,加并发只是往队列堆人。
2. **RT = 排队 + 处理。** 瓶颈满后,加并发 → 队列变长 → RT 线性涨。
3. **Little 定律:`并发 L = TPS × RT`。** TPS 封顶后,L 翻倍只能让 RT 翻倍。
   - 验证(并发256):`19745 × 0.013s ≈ 257 ≈ 256` ✓
4. **单行热点最惨**:瓶颈是一把锁,容量恒为 `1/RT_处理`,
   **加机器、加并发都扩不了容**,只能眼看 RT 爆炸、请求堆积、线程/连接耗尽、雪崩。
5. **高并发真正的危害是 RT 线性恶化,不是 TPS 掉**(理想纯内存锁下 TPS 是平的)。
   看 RT 比看 TPS 抖动更能反映过载。

## 四、两类解法

| 方案 | 本质 | 效果 | 局限 |
|------|------|------|------|
| 热点更新技术(文章) | 压小锁占用时间 = **压小 RT** | 把天花板 `1/RT` **抬高** | 仍是单行,**突破不了天花板**;挑版本(Percona 5.7.17)、仅单机 |
| **余额/库存拆分(分桶)** | 1 行拆成 N 行,1 把锁拆成 N 把 | 瓶颈容量 ×N,封顶线整体抬 N 倍,**真正绕开天花板** | 读取需聚合 N 桶;有热点桶倾斜问题 |

> 支付/直播打赏里的"热点账户"(超级主播余额被百万人同时写),
> 业界主流就是**余额拆分**:把一个账户拆成 N 个子账户分散写入。

## 六、分桶对照实验(`bucket/`,验证"余额拆分")

固定并发=256,只改桶数(把 1 行拆成 N 个各带锁的桶),机器 12 核:

```
桶数    实测TPS      平均RT      相对单桶
1      19722      13.0ms      1×    ← 等于单行,封顶
2      39556      6.5ms       2×
4      78636      3.3ms       4×
8      149650     1.7ms       ~7.6×   线性段:拆 N 倍 ≈ 快 N 倍
16     201712     1.2ms       ~10×  ← 涨幅放缓
32     197149     1.16ms      ~10×  ← 平了
64     198436     1.13ms      ~10×  ← 平了
```

- **线性段(1→8)**:TPS 按桶数翻倍、RT 反向减半 → 拆分把天花板抬高 ~N 倍,
  这正是 hotrow 单行那条死线(~20000)突破不了、而拆分能突破的原因。
- **撞墙段(16→64)**:TPS 封顶 ~200000。忙等吃 CPU,12 核最多并行 12 个临界区,
  锁瓶颈解开后撞上 **CPU 核数**(`12 × ~16700 ≈ 200000`)这个新瓶颈。

> **最核心的一句:瓶颈只会转移,不会消失。** 拆分解开"单行锁" → 撞上"CPU 核数";
> 真实 DB 还会接着撞 磁盘IO / binlog刷盘 / 网络。性能优化 = 不断定位并搬走当前瓶颈。

运行:`go run ./hotrow/bucket`

## 七、真实库存场景:借库存 + 合并对账(`stock/`)

分桶不是免费的。把库存拆成 N 个桶分散扣减后,会冒出两个新问题,真实系统必须处理:

### 问题:库存倾斜 → 少卖

请求分布通常不均(按 user_id/商品 hash 天然倾斜)。热门桶先卖空,
**冷门桶还有货**。若只认"本桶",打到空桶的请求会被误报"售罄" —— 明明有货却没卖出去,
这就是**少卖**。

### 两个必备步骤

1. **借库存(rebalance / 匀桶)**:本桶空了,扫描其他桶,从有货的桶里扣。
   兜底到"真的全空"才报售罄。
2. **合并对账(merge)**:卖完把所有桶剩余**求和归集**,核对 `售出 + 剩余 == 初始`。
   既验证守恒(防超卖/凭空多扣),又得到对外的权威总数。

### 实测(总库存 10000,8 桶,60% 请求倾斜到 0 号桶)

```
策略A 仅本桶:  售出 9970,各桶剩余=[0 0 0 0 0 0 22 8] → ❌ 少卖 30 件(有货却报售罄)
策略B 借库存:  售出 10000,各桶剩余=[0 0 0 0 0 0 0 0] → ✅ 卖光,不少卖
两者都满足:   售出 + 合并剩余 == 10000  → 桶锁内校验 val>0,绝不超卖
```

### 三条底线规律

| 风险 | 成因 | 对策 |
|------|------|------|
| **超卖** | 多请求同时扣同一桶 | 每桶扣减放在**桶锁内**校验 `val>0`(demo 已保证,守恒成立) |
| **少卖** | 库存倾斜,冷桶有货被误报售罄 | **借库存**:本桶空了去别的桶扣 |
| 对不上账 | 拆开后不知道总剩余 | **合并归集**:求和所有桶,核对守恒 |

> 映射到支付/账户:热点账户余额拆成 N 个子账户分散写(对应分桶);
> 扣款时某子账户不足要能跨子账户兜底(对应借库存);
> 查总余额/对账时把 N 个子账户**聚合求和**(对应合并)。这三步缺一不可。

运行:`go run ./hotrow/stock`

## 八、落到 DB:那套 MySQL 私有语法在 Go 里怎么写(`dbdeduct/`)

腾讯云文章里的 `COMMIT_ON_SUCCESS / ROLLBACK_ON_FAIL / QUEUE_ON_PK / TARGET_AFFECT_ROW`
是**腾讯云 Percona 定制内核的私有语法**,标准 MySQL 不认识。Go 的 `database/sql`
不解析 SQL 方言,所以:

- **库支持那套语法** → Go 里没有特殊写法,把 SQL 当普通字符串 `db.Exec` 即可。
- **标准 MySQL** → 用**单条原子条件更新 + 看 RowsAffected** 达成同样目的(推荐):

```go
res, _ := db.ExecContext(ctx,
    `UPDATE stock SET num = num - ? WHERE id = ? AND num >= ?`, n, id, n)
affected, _ := res.RowsAffected()
// affected==1 → 扣减成功;==0 → 库存不足(改0行,无副作用,无需回滚)
```

这条语句在 autocommit 下:**取行锁 → 改 → 提交 → 放锁,一次往返,中间无应用逻辑**,
正是热点更新"最小持锁"的目的。关键字逐一映射:

| 私有关键字 | 标准 MySQL + Go 等价物 |
|-----------|----------------------|
| `COMMIT_ON_SUCCESS` | autocommit 单语句,成功自动提交 |
| `ROLLBACK_ON_FAIL` | `WHERE num>=?` 不命中 → 改 0 行、无副作用(天然回滚) |
| `TARGET_AFFECT_ROW 1` | Go 里判断 `RowsAffected()==1` |
| `QUEUE_ON_PK id` | `WHERE id=?` 走主键,InnoDB 行锁天然在该行排队 |

⚠️ 坑:若放进事务做"扣库存+建订单",行锁会从 UPDATE 持有到 COMMIT,期间应用逻辑
都在持锁中 → 又退化成长持锁。对策:**扣库存放事务最后一步、事务体尽量小**。
(这正是那个私有特性存在的理由——避免跨应用逻辑持有热点行锁。)

运行:`go run ./hotrow/dbdeduct`(无 DSN 时只打印示例 SQL)

## 九、秒杀最常用:Redis + Lua 原子扣减(`redislua/`)

Redis 单线程执行,**一段 Lua 脚本整体原子**,中途不会插入别的命令。于是
"读库存→判断够不够→扣减"合成一次原子操作、一次往返:既不超卖(对比 GET 再 DECR
有竞态),又把热点全挡在内存里、不压 DB。这是前面所有思想搬到内存的终极形态。

```lua
-- 单 key 原子扣减:KEYS[1]=库存key, ARGV[1]=数量。返回 1成功/0不足/-1不存在
local stock = redis.call('GET', KEYS[1])
if stock == false then return -1 end
if tonumber(stock) < tonumber(ARGV[1]) then return 0 end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1
```

```go
// go get github.com/redis/go-redis/v9
n, err := rdb.Eval(ctx, luaDeduct, []string{"stock:1001"}, 1).Int()
// n==1 成功 / n==0 库存不足 / n==-1 不存在
// 生产建议 redis.NewScript(lua):首次 EVAL,之后走 EVALSHA 省带宽
```

更妙的是:前面 `stock/` 里手写的"借库存"逻辑,用一段**多 key Lua** 就能原子完成:

```lua
-- 分桶+借库存:KEYS=N个桶key, ARGV[1]=数量。返回命中的桶号(1..N),全空返回0
for i = 1, #KEYS do
  local s = redis.call('GET', KEYS[i])
  if s ~= false and tonumber(s) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[i], ARGV[1])
    return i
  end
end
return 0
```

实测(库存1万,并发256,请求2.5万):两种方式都 `售出+剩余==10000`,绝不超卖。
> 注意:Redis 扣减成功后需**异步、可靠地落库**(消息队列 + 对账补偿),Redis 只是
> 挡洪峰的前置库存,DB 仍是最终账本。

运行:`go run ./hotrow/redislua`(内存模拟 Redis 原子语义,无需装 Redis)
